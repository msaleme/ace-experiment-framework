## Understanding the 7 Required Elements

Every experiment in ACE has these 7, non-negotiable components:

### 1. **Immutable Baseline Snapshot**

A frozen reference point for comparison.

**Example baseline:**
```yaml
baseline_id: baseline_gpu_transformer_bf16_v1
code_commit: "abc123def"
compiler_version: "cudnn-8.9.0"
environment: {CUDA_VISIBLE_DEVICES: "0"}
created_timestamp: "2026-03-27T09:00:00Z"
```

**Why immutable?**
- Prevents "moving the goalposts"
- Baseline drift invalidates experiments
- Once created, cannot be modified

**Location:** `baselines/*.yaml`

---

### 2. **Declared Mutation Scope**

Explicit, written commitment to what can change.

**Example:**
```yaml
mutation_scope:
  - quantization_config       # ✓ ALLOWED
  - fallback_policy           # ✓ ALLOWED
  - layer_fallback_thresholds # ✓ ALLOWED

# These are NOT allowed:
no_mutation_allowed:
  - model_architecture        # ✗ FORBIDDEN
  - training_data             # ✗ FORBIDDEN
```

**Why declare?**
- Prevents uncontrolled experiments
- Forces focus on hypothesis
- Maintainable and reproducible

**Enforced by:** `ExperimentRunner.verify_mutation_scope()`

---

### 3. **Repeated Trials**

Multiple runs with different random seeds to rule out luck.

**Configuration:**
```yaml
trials: 10          # Minimum often needed
seeds:
  - 42
  - 101
  - 2022
  - ... (8 more)
```

**Requirements:**
- Minimum 10 trials (configurable)
- Different seed each time
- Variance expected and normal
- Outliers detected and reported

**Why repeated?**
- Single run can be lucky (or unlucky)
- Variance is information, not noise
- Statistical rigor demands replication

**Collected by:** `MetricsCollector.record_trial()`

---

### 4. **Quality Floor**

Hard constraint: solution must maintain quality.

**Example:**
```yaml
quality_floor:
  metric: accuracy
  minimum_relative_to_baseline: 0.99  # 99% of baseline
  absolute_minimum: 0.90              # Never below 90%
```

**Enforced by:** `StatsEvaluator.check_quality_floor()`

**Why hard constraint?**
- Speed is worthless if it's wrong
- Quality loss must be intentional and bounded
- Non-negotiable for production systems

---

### 5. **Holdout Benchmark Support**

Benchmarks never seen during optimization.

**Three benchmark sets:**
```
Development   (used for tuning)        → Use in outer loop
  ↓
Validation    (test transferability)   → Use for validation
  ↓
Holdout       (NEVER OPTIMIZED)        → Final test only
```

**Why?**
- Development benchmarks can be overfit
- Validation tests transfer to similar workloads
- **Holdout proves it generalizes to unknown data**

**Implemented as:** `BenchmarkRegistry.get_benchmark_set("holdout")`

**Code prevents optimization on holdout:**
```python
if benchmark_id in holdout_set:
    raise ValueError("Cannot optimize on holdout benchmarks!")
```

---

### 6. **Statistical Verdict**

Rigorous, automatic pass/fail based on pre-defined rules.

**Required for acceptance:**
1. ✓ Effect size significant (Cohen's d ≥ 0.10)
2. ✓ Statistically significant (p < 0.05)
3. ✓ Confidence interval excludes zero
4. ✓ Quality floor maintained
5. ✓ Latency budget met
6. ✓ Holdout validation passed
7. ✓ Complexity overhead < 5% of savings

**Automatic verdict:**
```
All conditions met  → ACCEPTED
One or more fail    → REJECTED
Borderline/noisy    → INCONCLUSIVE
```

**Generated by:** `StatsEvaluator.generate_verdict()`

---

### 7. **Archived Report Output**

Complete, reproducible documentation of everything.

**Generated artifacts:**
- `experiment.json` - Complete record, machine-readable
- `trials.json` - Raw trial data for replay
- `report.md` - Human-readable summary
- `comparison.md` - vs. baseline and other experiments
- `dashboard.md` - Portfolio overview

**Location:** `reports/exp_<id>/`

**Includes:**
- Hypothesis and motivation
- Baseline reference (immutable)
- Mutation details applied
- All trial metrics
- Statistical analysis
- Reproducibility metadata (code commit, dependencies, seeds)

## Running Your First Experiment

### Step 1: Define the baseline

**File:** `baselines/baseline_my_experiment.yaml`

```yaml
baseline_id: baseline_custom_v1
hardware:
  name: A100-40GB
code_commit: "abc123"
compiler_version: "cudnn-8.9.0"
# ... (full environment snapshot)
```

### Step 2: Create benchmark definitions

**Files:** `configs/benchmarks/*.yaml`

Three sets required:
- Development (optimize on this)
- Validation (test transfer)
- Holdout (final test, READ-ONLY)

```yaml
benchmark_id: my_benchmark_dev
benchmark_set: development  # or "validation" or "holdout"
```

### Step 3: Write experiment config

**File:** `experiments/near_term/exp_XXX_my_experiment.yaml`

**Must include all 7 elements:**
```yaml
experiment_id: exp_XXX_my_experiment
hypothesis: "..."
baseline: baseline_custom_v1          # 1. Immutable baseline
mutation_scope: [...]                 # 2. Declared scope
trials: 10                            # 3. Repeated trials
evaluation_config:
  quality_floor: 0.99                 # 4. Quality floor
  require_holdout_pass: true          # 5. Holdout validation
  # ... acceptance criteria for 6. Statistical verdict
reporting:                            # 7. Archived reports
  # ... report config
```

### Step 4: Implement trial executor

```python
def trial_executor(trial_num: int, benchmark_id: str):
    """Run one trial of the experiment."""
    # 1. Load baseline model
    # 2. Apply mutation
    # 3. Run benchmark
    # 4. Measure metrics
    # 5. Return metrics dict
    return {
        "accuracy": accuracy_value,
        "latency_ms": latency_value,
        "energy_joules": energy_value,
        # ... all primary metrics
    }
```

### Step 5: Run experiment

```python
from src import ExperimentRunner

runner = ExperimentRunner(baseline_mgr, benchmark_registry, metrics_collector)
experiment = runner.run_experiment(
    experiment_id="exp_XXX_my_experiment",
    trial_executor=trial_executor,
)
```

### Step 6: Check verdict and reports

```python
print(f"Verdict: {experiment.verdict}")

# Reports generated automatically:
# - results/exp_XXX_my_experiment/
# - reports/exp_XXX_my_experiment_report.md
```

## Key Principles

### Immutability is Enforced
```python
# This raises an error - baselines cannot be changed
baseline_mgr.create_baseline(
    baseline_id="baseline_v1",  # Already exists
    # ...
)
# ValueError: Baseline baseline_v1 already exists. Baselines are immutable.
```

### Mutation Scope is Validated
```python
# This checks that only declared changes were made
runner.verify_mutation_scope(
    experiment_id="exp_001",
    changed_items=["quantization_config", "fallback_policy"],
)
# If any undeclared items changed, raises error
```

### Holdout Benchmarks are Protected
```python
# This prevents optimization on holdout
if benchmark_id in holdout_set:
    raise ValueError("Cannot optimize on holdout benchmarks!")
```

### Statistical Rigor is Mandatory
- No single-run acceptance
- Effect size + significance required
- Confidence intervals computed
- Outliers detected and reported

## Understanding Verdicts

### ACCEPTED ✓
```
All conditions met:
- Effect size >= 0.10 (Cohen's d)
- p < 0.05 (Welch's t-test)
- Quality >= 99% of baseline
- Latency within budget
- Holdout validation passed
- Complexity overhead < 5%

Action: Archive and branch to next experiments
```

### REJECTED ✗
```
One or more conditions failed:
- Insufficient effect size
- Not statistically significant
- Quality degradation
- Latency budget violated
- Holdout validation failed
- Overhead too high

Action: Update hypothesis and try different approach
```

### INCONCLUSIVE ?
```
Too noisy or insufficient data:
- High variance (CV > 20%)
- Borderline significance (0.03 < p < 0.10)
- Too few successful trials (< 8 of 10)
- Conflicting metrics

Action: Increase trials or investigate sources of noise
```

## Common Patterns

### Pattern 1: Simple optimization loop

```python
# Load experiment from config
experiment = runner.load_experiment_config(
    Path("experiments/near_term/exp_001.yaml")
)

# Run with your trial executor
experiment = runner.run_experiment(
    experiment_id=experiment.experiment_id,
    trial_executor=my_executor,
)

# Verdict is automatic
if experiment.verdict == Verdict.ACCEPTED:
    print("Success! Moving to next experiment")
```

### Pattern 2: Batch execution

```python
# Run multiple experiments in sequence
for exp_config in Path("experiments/near_term").glob("*.yaml"):
    experiment = runner.load_experiment_config(exp_config)
    experiment = runner.run_experiment(
        experiment_id=experiment.experiment_id,
        trial_executor=trial_executor,
    )
    results_store.save_experiment(experiment)

# Generate portfolio summary
dashboard = report_generator.generate_portfolio_dashboard(
    results_store.list_experiments()
)
```

### Pattern 3: Branching from accepted results

```python
# After exp_001 is accepted, propose exp_002
parent = results_store.load_experiment("exp_001")

if parent.verdict == Verdict.ACCEPTED:
    # Create child experiment with refined hypothesis
    child = runner.create_experiment(
        parent_experiment_id="exp_001",
        hypothesis_text="Further refinement of INT4...",
        # ...
    )
    # Run and validate child
    child = runner.run_experiment(...)
```
